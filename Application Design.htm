Theme & Layout

In computing, a theme is a set of graphical appearance details. A theme usually comprises colors and shapes for windows and graphical control elements. Themes are introduced to customize the look and feel of a computer software or even of an operating system.

Applications usually use standard themes provided by an operating system. Accordingly, applications are more or less similiar regarding layout of the application's windows. The figure below shows layouts of MS Notepad and Notepad++, two text processor applications, on Windows 10 operating system.

Fig. 1. Classical Windows applications.

The main reason for implementation of specific application theme was to introduce something new and to breach monotony of windows’ applications attracting attention of the user.

Making Layout More Attractive

It is well known that power system analyses, especially transient stability, is fair complex since it involves modelling of lines, transformers, loads and generating units. Generating units comprise generators, prime movers (most often turbines) and control systems such as turbine governors and automatic voltage regulators. Today, power engineers decide to profile themselves for electricity market rather than power system analysis since electricity market expertize does not require deep and complex knowledge. In addition, computer applications for power system analysis are pretty much complex without unique user approach. For instance, some of the applications require definition of new types of power system elements (lines, transformers, generators etc.) or usage of already defined ones in order to enter particular power system element into model while the user interface of another software is based on direct input of data for each element. On the other hand, some software use data in per unit system while the others use absolute values. Particular issue is construction of control systems required for transient stability analyses. Almost every software has its own specific approach starting from graphical builders up to using of particular programming or simulation language.

Due to all aforementioned trainings are organized for each particular software with the software user interface (UI) emphasized.

Different power system analysis software are used to carry out system studies containing power system calculations depending on licenses and requirements of system operators to which systems the studies are referred to.

Bearing in mind all aforementioned about different power system software, it is somehow reasonable why engineers choose rather to deal with electricity market issues rather than power system analysis. Accordingly, this results in lack of experts in power system analysis field.

It seems software companies have developed their software fairy complex in order to make their usage almost impossible without appropriate training. The proof for this assertion is the fact when a user completes training for a particular software he is not capable another one due to, at the first place, significant differences of UIs of the software. In addition, software companies make business by conducting trainings for their software.

Power engineer should be capable of using UI intuitive power system analysis software with the help of adequate application manual. Trainings should be oriented to theoretical consideration of complex power system elements such as generators, turbines, excitation systems etc. and physical phenomena in order to draw clear parallel between mathematical models and devices. Such trainings would be far more educational than those oriented to usage of UI of particular application.

Frontal Approach

Frontal approach means all GUI commands, which are used by user in 99% of cases, are brought to the application front. This means user can access these commands by single mouse click or tap, depending on used device, without switching between tabs or opening dialog windows. Such approach makes application usage straightforward and minimizes number of actions undertaken by user.

As mentioned before MS Ribbon was used to implement the application GUI. MS Ribbon is used to implement GUI of almost all state-of-the-art Windows desktop applications. Layout of the application ribbon is shown in the next figure.

Besides mouse clicking on ribbon buttons the application commands can be executed by using accelerator keys, i.e. keyboard shortcuts. Usage of accelerator keys is the fastest way to execute application commands. Most of desktop applications have accelerator keys defined. Some accelerator keys are standardized such as: Ctrl+C - Copy, Ctrl+V - Paste, Ctrl+O - Open, Ctrl+S - Save, Alt+Tab - Switch to next application, etc. Table containing accelerator keys is given in the application Help.

Windows 10 spreads the idea that applications should bring content, i.e. working material, forward as much as it is possible so that it can easily be seen and accessed by user. In order to propagate this approach Microsoft made windows borders 1 px wide under Windows 10 which was not case under previous Windows versions where widows borders were 6 px wide. Explanation for window layout change is that content should be emphasized.

There are no general rules how to bring content forward due to different types of applications. Regarding power system simulators, content is pretty much complex since it is comprised of electrical data of power system elements (generators, lines, transformers, etc.) and, in most of cases, single line diagram (SLD). SLD is the most accepted way of power system visual representation. It provides:

-       Overall picture of entire power system at first glance;

-       Information about status (on/off) of each particular element as well as its load;

-       Fast switching actions (turn on/off) of any power system element;

-       Display of calculation results: voltages and power flows.

SLD could be found in any power dispatching center. It is almost impossible to think of real time transmission or distribution system control without having SLD. On the other hand, regarding offline analyses of power system, results of calculation are usually presented on SLDs because it provides the best overall picture of system power flows. However, dimensions of SLDs could be large depending of number of power system elements. For instance, entire walls in dispatching centers are used for SLD. Various techniques are used in offline power system simulators to bring entire SLD content to user on classic monitors such as: navigation, zooming, layering, etc.

It could also be seen in dispatching centers that system frequency is brought forward because it is global indicator about system operation. Any misbalance between production and consumption in entire power system results in system frequency deviation. Constant system frequency is one of power system control targets. Bearing in mind aforementioned, it is deemed to customize the application layout in a way to have system frequency displayed on the front.

The most often used system calculation is load flow. Load flow calculation provides starting point for transient stability, which is mandatory, and short circuits calculations which is optional. Results of load flow calculation is used for short circuits calculations if complete method is used.

Almost all of power system simulators display convergence of load flow calculation. Load flow convergence shows several issues:

-       Whether calculation converged or not;

-       Number of iterations. This information is significant because ill-conditioned power system models take higher number of iterations to converge;

-       Node with the largest power mismatch in particular and last iteration which is useful for model tuning if needed;

-       So-called outer loops in which PV nodes (generator nodes) are transferred into PQ nodes or PQ nodes, which previously were PV nodes, are transferred back into PV if certain conditions are met. This is well known backward-looking algorithm.

A Status bar is a horizontal window at the bottom of a parent window in which an application displays various kinds of status information, progress of a performing process or thread, hosts other child windows, etc. The status bar is usually divided into parts in order to provide more than one function. Most of Windows application have status bar implemented. Implementation of status bar depends on type of application. In our particular case status bar has one function that is specific only for this application: one of its parts is used to keep visual minimized child windows without reducing client (working area) of the application. This is very unique implementation of the status bar that cannot be found in any other application. More about such implementation of the status bar could be found in the application Help and Manual.

Bearing in mind everything aforementioned regarding the application front we come to proposal of its recommended layout, i.e. what should be brought to the application front:

1.     Ribbon;

2.     SLD and its navigation. Navigation of SLD is implemented as separate child window;

3.     Child window showing convergence progress of load flow;

4.     Child window showing system frequency;

5.     Status bar.

The ribbon has three tabs: File, Home and View. Tab File is always used for file operations which is standard in each ribbon implementation. Details of tabs File and View could be found in the application Help and Manual. Focus of this section is tab Home which contains all commands that are executed by user in most of cases. First two groups of commands, "Model Elements and Calculation Parameters” and “Areas and Regions", are related to editing by using flex grids (spreadsheets). Details about editing elements over flex grids are given in the application Help. Next two groups of commands, "Power Flow", "Faults" and "Transient Stability" contain buttons for launching/stopping system calculations: power (load) flow, short circuits and transient stability. Next two groups of commands in tab Home are "Power Flow Reports" and "Tools".
